
# Codex Workflow

## Purpose

This document defines how Codex should be used for this repository.

It is an internal workflow note for AI-assisted development.
Its goal is to make implementation more disciplined, reduce architectural drift, and prevent the repository from turning into a pile of loosely connected autogenerated code.

This file is intentionally operational.
It is not public-facing documentation.

This file should be read together with:

- `ASSUMPTIONS.md` = registry of ambiguity and simplifications

If there is a conflict between raw implementation convenience and documented architecture, the documented architecture should usually win unless there is an explicit decision to revise it.

## Core philosophy

Codex should be used as a disciplined implementation partner, not as an unsupervised code generator.

The project should be developed through:
1. specification,
2. local architectural decisions,
3. narrow implementation tasks,
4. tests,
5. review and refinement.

The most important principle is:

**Codex should implement bounded slices of the design, not invent the entire design on its own.**

---

## Primary development rule

Do **not** ask Codex to “build the whole project” or “implement the entire game from the PDF”.

Instead, ask it to perform small, concrete, testable tasks that fit into the documented architecture.

Preferred granularity:
- one module,
- one subsystem,
- one vertical slice,
- one rule family,
- one loader,
- one set of tests,
- one refactor with clearly stated intent.

This rule exists because large open-ended prompts tend to produce:
- invented assumptions,
- premature abstractions,
- duplicated logic,
- architecture drift,
- brittle code that is hard to trust.

---

## Role boundaries

### What Codex is good for in this project

Codex is especially useful for:
- creating project skeletons inside an already chosen architecture,
- implementing dataclasses / enums / typed models,
- writing parsing and loading utilities,
- implementing narrow rule resolvers,
- generating tests from explicit expected behavior,
- refactoring modules for clarity,
- adding logging, serialization, and CLI wrappers,
- wiring together already-designed components.

### What Codex should not decide alone

Codex should **not** be allowed to silently decide:
- ambiguous rule interpretations from the rule PDF,
- the overall action abstraction without guidance,
- the RL reward design,
- macro-action vs micro-action architecture unless already decided,
- long-term package structure changes,
- hidden-information semantics,
- whether new abstractions are actually justified.

If a task touches one of these areas, the prompt should explicitly constrain the decision or instruct Codex to document assumptions instead of improvising.

## Standard workflow for adding a feature

When adding any nontrivial feature, the preferred order is:

1. Check relevant docs.
2. Determine which thread role should own the work under
   `orchestration_policy.md`.
3. Create or update a local thread report if the task is nontrivial.
4. Identify whether the change affects public spec or only implementation.
5. Update docs if needed.
6. Ask Codex for a plan or for narrowly scoped code changes.
7. Make Codex implement the smallest useful slice or delivery package.
8. Ensure tests exist or are added.
9. Review resulting structure for architectural drift.
10. Update `ASSUMPTIONS.md` if a rule interpretation or simplification was introduced.
11. Decide whether the result should become one commit or several under
    `commit_policy.md`.

Local thread reports should live under `docs/internal/thread_reports/`.
They are useful for scratch planning and handoff, but they must not become the
only place where durable decisions live.

### Minimal example

Bad:
- “Implement the whole simulator.”

Better:
- “Implement axial hex coordinates, neighbors, and unit tests.”
- “Add `UnitType`, `TerrainType`, and `ActionType` enums.”
- “Implement mission loading for a single mission config.”
- “Implement legal movement generation only.”
- “Add deterministic RNG wrapper and tests.”

## Preferred prompt patterns

### Pattern 1 — architecture-first prompt
Use when starting a subsystem.

Example intent:
- inspect existing repository layout,
- propose a minimal module plan,
- do not implement everything yet,
- respect `repo_layout.md` and public docs,
- identify missing pieces and risks.

This is useful before writing code in a new subsystem.

### Pattern 2 — narrow implementation prompt
Use for most feature work.

Structure:
1. name the target files,
2. describe the feature narrowly,
3. state constraints,
4. require tests,
5. forbid unrelated refactors.

Example intent:
- implement only hex neighbor logic in `domain/hexgrid.py`,
- add tests in `tests/test_hexgrid.py`,
- do not add RL code,
- do not refactor unrelated modules.

### Pattern 2B — delivery-package prompt
Use when one Delivery Thread should carry a bounded package across multiple
turns.

Structure:
1. name the package and the package boundary,
2. define substeps/checkpoints,
3. define stop conditions,
4. define verification expectations,
5. require a structured checkpoint report back to the user.

This is preferred over opening a new thread for every micro-stage.

### Pattern 3 — refactor prompt
Use when improving structure.

Structure:
1. explain the structural problem,
2. state which behavior must remain unchanged,
3. require tests to keep passing or be updated,
4. forbid architecture expansion beyond the refactor goal.

### Pattern 4 — analysis-before-edit prompt
Use when uncertain.

Ask Codex first to:
- inspect relevant files,
- summarize current structure,
- identify risks,
- propose a plan,
- wait before implementing.

This is especially useful for larger changes.

## Prompting rules for this repository

### 1. Always anchor Codex to the docs
For nontrivial tasks, prompts should reference the relevant design docs, for example:
- `docs/internal/repo_layout.md`
- `docs/architecture.md`
- `docs/state_model.md`
- `docs/action_model.md`
- `ASSUMPTIONS.md`

This reduces improvisation.

### 2. Explicitly state what is out of scope
Almost every prompt should include lines such as:
- do not add RL-related code,
- do not introduce new package layers,
- do not change public interfaces unless necessary,
- do not implement unrelated rules,
- do not refactor the whole repository.

### 3. Prefer one decision at a time
If a task touches both:
- architecture,
- implementation,
- tests,
- docs,

then break it into stages unless the change is very small.

### 4. Require justification when structure changes
If Codex wants to introduce:
- a new package,
- a new abstraction,
- a base class,
- a protocol/interface,
- a new config layer,

it should be asked to justify why that is better than a simpler direct implementation.

### 5. Require tests for rule logic
If the task adds or changes game-rule behavior, the prompt should usually require tests.

### 6. Keep scripts thin
If Codex writes code in `scripts/`, the prompt should remind it to keep most logic inside `src/`.

---

## Decision policy for ambiguous rules

If Codex encounters ambiguous or underspecified rules, it should not quietly invent a definitive interpretation.

Preferred behavior:
1. implement the narrowest reasonable interpretation,
2. surface the assumption explicitly in code comments if needed,
3. record the simplification or ambiguity in `ASSUMPTIONS.md`,
4. avoid overcomplicating the engine to handle speculative possibilities.

The project values explicit simplification over hidden guesswork.

---

## Rules for working with tests

### Test-first preference
For especially tricky rule logic, it is often preferable to ask Codex to write or outline tests before implementing the code.

### Test scope
Tests should cover:
- deterministic behavior,
- legality,
- transition correctness,
- reproducibility,
- config loading,
- regression-sensitive behavior.

### Anti-patterns
Avoid prompts that let Codex:
- remove tests because implementation is difficult,
- weaken assertions without reason,
- change behavior and silently rewrite tests to match it,
- add superficial tests that do not check the actual contract.

### Good prompt add-on
Useful wording:
- “Add focused tests that verify the documented behavior, not just smoke tests.”

---

## Rules for working with documentation

Codex may edit docs, but documentation edits should be intentional.

### When docs should be updated
Update docs when a change affects:
- architecture,
- file responsibilities,
- state structure,
- action abstraction,
- assumptions,
- scenario/config format,
- reproducibility guarantees.

### Internal vs public docs
- Internal docs are for working process and local planning.
- Public docs are for durable project truth.

Stable project behavior should not live only in internal files.

---

## Specific anti-patterns to avoid

### 1. Premature RL scaffolding
Do not ask Codex to build:
- Gym wrappers,
- reward systems,
- policy code,
- training loops,
- evaluation dashboards,

before the engine and baseline simulation loop are real and tested.

### 2. Empty architectural expansion
Avoid creating many empty modules “for future use” unless the structure is immediately helpful.

### 3. Deep inheritance for no reason
Prefer explicit functions and small dataclasses to elaborate class hierarchies unless there is clear repeated behavior.

### 4. Duplicate rule logic in agents
Agents should choose actions, not re-encode the game engine.

### 5. Hiding assumptions in code
If something is a simplification, document it.
Do not let it disappear inside implementation details.

### 6. Notebook-driven architecture
Exploration in notebooks is fine.
Core behavior in notebooks is not.

## Examples of good task decomposition

### Example A — early engine setup
1. create package skeleton under `src/solo_wargame_ai/`
2. add enums
3. implement `hexgrid.py`
4. add tests for hex neighbors
5. implement terrain model
6. implement unit dataclasses
7. implement `GameState`

### Example B — movement subsystem
1. define movement-related action model
2. generate legal movement actions
3. implement movement resolution
4. test movement legality and resulting state updates

### Example C — mission loading
1. define mission config format for one scenario
2. implement parser/loader
3. validate loaded objects
4. test one deterministic sample mission

### Example D — first playable slice
1. mission load
2. initial state creation
3. legal actions for first decision point
4. apply one action
5. produce text log
6. run deterministic short trace

This style is strongly preferred over giant monolithic implementation requests.

## Review checklist after Codex changes

After Codex finishes a task, review the result with these questions:

1. Did it stay within scope?
2. Did it invent assumptions not requested?
3. Did it respect package boundaries?
4. Did it add unnecessary abstractions?
5. Did it put rule logic in the right place?
6. Did it add or preserve meaningful tests?
7. Did it silently change behavior outside the target feature?
8. Does the change still match `repo_layout.md` and public docs?
9. Should `ASSUMPTIONS.md` be updated?
10. Should any public documentation be updated?

If the answer to several of these is bad, prefer rollback or a narrower reimplementation.

## How to use Codex at different project stages

### Stage 0 — planning
Use Codex mostly for:
- repository inspection,
- architecture suggestions,
- file skeletons,
- doc drafting,
- identifying missing pieces.

### Stage 1 — engine foundation
Use Codex for:
- dataclasses,
- enums,
- grid code,
- state containers,
- loaders,
- resolvers,
- tests.

### Stage 2 — playable MVP
Use Codex for:
- CLI runner,
- replay/logging,
- mission validation,
- baseline agent wiring,
- deterministic scenario runs.

### Stage 3 — baseline agents
Use Codex for:
- random agent,
- heuristic agent,
- batch evaluation harness,
- metrics collection.

### Stage 4 — RL integration
Only after the above is stable.
Then use Codex for:
- observation encoding,
- action encoding,
- legal action masks,
- environment wrapper,
- reward prototypes.

### Stage 5 — experiments and refinement
Use Codex for:
- analysis scripts,
- benchmark reporting,
- experiment config handling,
- structured refactors.

## Preferred communication style with Codex

The most reliable interaction style is:
- specific,
- bounded,
- architecture-aware,
- test-aware,
- explicit about non-goals.

When in doubt, first ask for:
- a short plan,
- identified risks,
- the exact files to edit,
- and only then request implementation.

This is often better than asking for immediate code.

---

## Suggested reusable prompt fragments

These are not strict templates, but useful recurring constraints.

### Fragment: scope control
- Respect the current repository architecture and do not introduce unrelated abstractions.
- Keep the change narrowly scoped to the requested subsystem.
- Do not refactor unrelated files.

### Fragment: tests
- Add focused tests for the new behavior.
- Prefer deterministic tests with explicit expected outcomes.
- Do not replace strong assertions with weaker smoke tests.

### Fragment: architecture
- Keep domain logic inside `src/solo_wargame_ai/domain/`.
- Do not place rule logic in scripts, notebooks, or agent code.
- Keep the implementation compatible with the documented domain-first design.

### Fragment: ambiguity handling
- If a rule is ambiguous, implement the narrowest reasonable interpretation and note it clearly.
- Do not silently invent broad behavior not grounded in the current spec.

### Fragment: output discipline
- Explain what files you changed and why.
- Mention any assumptions or edge cases introduced.
- Mention whether public or internal docs should be updated.

---

## Escalation rule

If a task starts requiring decisions about:
- game abstraction,
- action-space design,
- hidden-information semantics,
- reward design,
- package reorganization,

then stop using purely implementation-style prompts.
Switch to analysis-before-edit prompts and make the architectural choice explicit first.

---

## Final principle

Codex is most valuable in this repository when it helps make the project:
- cleaner,
- faster to implement,
- better tested,
- more reproducible,
- and more faithful to the documented structure.

It is least valuable when it is allowed to improvise the project into an architecture that was never consciously chosen.

In short:
**specify first, constrain well, build in slices, test constantly, document assumptions.**
